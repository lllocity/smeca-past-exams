# CLAUDE.md (smeca-past-exams)

## プロジェクト概要
中小企業診断士試験（1次試験・全7科目）の過去問演習を効率化し、網羅的に学習するための個人用Webアプリケーション（日本語名：中小企業診断士 全科目対応・過去問演習アプリ）。スキマ時間の学習最適化と、科目・論点ごとの弱点克服を目的とする。

## 対象科目（全7科目）
1. 経済学・経済政策
2. 財務・会計
3. 企業経営理論
4. 運営管理
5. 経営法務
6. 経営情報システム
7. 中小企業経営・中小企業政策

## 技術スタック
- **Frontend/Framework:** Next.js (App Router), TypeScript, Tailwind CSS
- **Deployment:** Vercel
- **Backend/BaaS:** Supabase (PostgreSQL, Storage for images, Auth)

## 主要な要件・機能と運用フロー
1. **過去問演習機能:** 科目別の一問一答、即時正誤判定＆解説、セッション内のリアルタイム得点表示。スマホ最適化（レスポンシブUI）によるスキマ時間対応。
2. **データインポート機能:** 公式サイトから手動ダウンロードした過去問PDFを、事前にGemini（LLM）を用いてクリーンなJSONデータに変換。そのデータをバックエンド（BE）のインポートAPIを介してデータベースへ一括登録する仕組み。
3. **管理画面（手動修正エディタ）:** インポートした問題データのテキスト修正（LLMの変換ミスの調整など）や、経済学のグラフ・財務 of 計算表といった複雑な図表イメージ（PDFから手動で撮影したスクショ画像）を、後からドラッグ＆ドロップでSupabase Storageへアップロードして問題データに紐付けられる編集画面。問題文や解説の表示はMarkdown記法に対応させる。
4. **学習効率化機能:** 一度不正解になった問題を記録し、後から反復して解き直せる復習機能。回答時に「確信あり」「たぶん・勘」のフラグを付与し、「正解したけど勘だった問題」を復習対象に残せる「自信度フラグ」機能。
5. **分析・ログ:** 科目や論点（分野）ごとの進捗率・誤答傾向の分析機能。全体の進捗率、科目ごとの正解率、日ごとの解答数を可視化する総合ダッシュボード。

## 🧐 クロスレビュー（Cross-Review）＆壁打ちプロセス（最重要ルール）
Claude Codeが提案や実装を行う際は、必ず以下のクロスレビュー（多角的な整合性チェック）とプランモードでの合意形成プロセスを経て行うこと。

1. **プランモード（Plan Mode）での壁打ち徹底**
   - 勝手にコード生成・実装（実装モード）に入ってはならない。まずは必ずプランモード（`/plan`）にて、設計思想やアーキテクチャについてユーザーと壁打ちを行うこと。
   - 既存のDDLやJSONスキーマ案は「あくまで初期のたたき台」として扱い、より最適な設計（拡張性、将来的な集計のしやすさ、Supabase無料枠への配慮など）がないか、複数案を比較検討して提示すること。
2. **クロスチェック（多角的な要件検証）**
   - 新しい機能やテーブルを追加・修正する際は、既存の要件（一問一答、手動エディタ、インポートフローなど）に悪影響や矛盾が生じないか、全体の整合性をクロスレビューすること。
   - 特に「全科目網羅」というスケールに耐えられるか（経済学のグラフ画像、財務の計算表、法務の長文など、各科目の特性に対応できるか）を常に検証すること。
3. **レビュー観点（セルフチェックリスト）**
   - **RDBの正規化 vs 運用のシンプルさ**: `jsonb` 型や配列型の多用が、将来的な分析（「誤答傾向の分析」や「論点タグごとの集計」など）に不利に働かないか。
   - **インポートの堅牢性**: Geminiから出力されるJSONのパースエラーを防ぐためのバリデーションやスキーマ定義が強固か。
   - **パフォーマンス＆コスト**: 個人利用（Supabase/Vercel無料枠）の制限内に確実に収まるか。

## コーディング規約 / 指針
- TypeScriptの型定義を厳格に行うこと。
- コンポーネントは機能ごとに小さく分割し、軽量でレスポンシブな設計にすること。

## テストポリシー

**新機能・バグ修正を実装する際は、対応するユニットテストを必ず並行して作成すること。**

- テストフレームワーク: Vitest + React Testing Library
- テストファイルは `src/__tests__/` 以下に配置し、ソースのディレクトリ構造を踏襲する
  - `src/__tests__/api/` — API ルートのバリデーションロジック
  - `src/__tests__/lib/` — 純関数ユーティリティ
  - `src/__tests__/components/` — Client Component の状態遷移・分岐
- 純関数的なロジック（バリデーション・集計計算）はユニットテストで全分岐を網羅する
- Client Component の状態遷移・UI 分岐は React Testing Library でコンポーネントテストを書く
- Supabase クライアントは `vi.mock('@/lib/supabase/client')` でモックし、DB に依存しない形にする
- Server Component は Supabase 依存が強いためテスト対象外とする（ROI が低い）
- `npm run test` でテスト全件通過を確認してからコミットする
